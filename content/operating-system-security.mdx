Almost every interesting class of attack ends, eventually, at the operating system. The web shell wants to read `/etc/shadow`. The phishing payload wants to install a service. The exploit wants to flip a `cred` struct from `uid=1000` to `uid=0`. Whatever the entry vector, the operator's last mile is always the kernel — and the kernel, fortunately and unfortunately, has spent forty years building defences specifically against this last mile.

This essay is a tour of those defences. Not the *features* — every textbook lists the features — but the *boundaries they actually defend*, the *attack primitives that defeat them*, and the *practical operator's view* of what's left when you've got SYSTEM/uid 0 in your sights.

## The shape of the problem

Operating-system security is the discipline of saying *no*. No, this user-mode process may not read that page of memory. No, this thread may not execute this region. No, this pid may not signal that pid. No, this file may be opened only with that particular bit set.

Three classical components show up over and over.

  { label: "Reference monitor", tool: "the kernel itself" },

A *subject* (someone with intent), an *object* (something with state), and a *reference monitor* (the bouncer at the door). The reference monitor is the kernel — and specifically the smallest part of the kernel that *must* be trusted, the **Trusted Computing Base** (TCB). The whole game of OS security is keeping the TCB small, correct, and isolated from the subjects it polices.

The textbook bouncer has three properties:

- **Tamper-proof.** Subjects can't reach in and modify the bouncer.
- **Always-invoked.** Every access goes through it; no side door.
- **Verifiable.** You can convince yourself, with bounded effort, that it's correct.

When an operator finds a privilege escalation, **at least one** of those three has failed.

## Privilege rings: the hardware boundary

At the lowest layer, the CPU itself enforces a privilege boundary. On x86 there are four rings (0–3). In practice everyone uses two of them: ring 0 for the kernel, ring 3 for user-mode. The CPU refuses to execute privileged instructions, touch certain MSRs, or access kernel memory unless the current ring permits it.

Crossing rings is expensive *by design* — the CPU saves and restores state on a context switch — but the protection it provides is the bedrock for everything else. A user-mode process *cannot* directly read kernel pages, *cannot* mask interrupts, *cannot* re-program the IDT. To do anything privileged it has to ask the kernel, and the kernel decides.

That asking happens through one mechanism: **the system call**.

## The system call boundary

A syscall is the only legitimate way for ring 3 to enter ring 0. On modern x86-64 Linux it's the `SYSCALL` instruction; on Windows it's `SYSCALL` too (or `int 0x2e` on legacy paths). The CPU jumps to a fixed, kernel-controlled handler stored in `IA32_LSTAR`. From the operator's perspective, this is the entire interface to the kernel:

<Terminal title="syscall — the whole interface, on Linux x86-64">
{`# call number in rax, args in rdi, rsi, rdx, r10, r8, r9
# x86-64 Linux

mov  rax, 0x3b      ; SYS_execve
mov  rdi, &"/bin/sh"
xor  rsi, rsi        ; argv = NULL
xor  rdx, rdx        ; envp = NULL
syscall              ; ← ring transition`}
</Terminal>

There are roughly 350 of them on Linux and a few thousand on Windows (counting both Nt* and Win32k). Every one of them is a potential bug. Every one of them is potential attack surface. *This is the largest bounded interface in computing*, and most of OS security is the engineering required to keep it correct.

Three properties matter to an operator:

1. **Argument validation.** Kernel routines have to assume every byte from user-mode is hostile. Pointer arguments must be validated with `copy_from_user`/`ProbeForRead`; struct sizes have to be re-checked; integer overflows in length fields have to be considered. Forgetting one is how `CVE-2016-5195` (Dirty COW) and a thousand others happened.

2. **State machines.** Many syscalls expect to be called in a certain order. `bind` then `listen` then `accept`. Reordering them, or calling them concurrently from two threads on the same fd, is how race conditions become exploits. *Most modern Linux LPEs are races.*

3. **Speculation.** As of 2018 we know the CPU can leak data across the ring boundary by speculatively executing past the syscall mode-switch. Mitigations are now permanent (KPTI, IBRS, retpoline, eIBRS) and they are very expensive.

The order matters. **DEP** killed code injection on the stack, so attackers invented **return-oriented programming**. ROP eats DEP for breakfast — so we got **ASLR** to take the addresses away. Info leaks beat ASLR — so we got **stack canaries** in the prologue. Stack canaries can be brute-forced over `fork()` — so we got **KASLR** to randomise the kernel itself. KASLR can be defeated by `prefetch` timing — so we got **KPTI** to remove kernel pages from the user view entirely. KPTI is expensive — so we paired it with hardware **CET** (shadow stack + indirect branch tracking) on modern CPUs.

Defence in depth means: when you find one bypass, the next layer is still there.

> An operator's mental model: every mitigation is a *cost*, not a *wall*. Stack canaries cost an info-leak; KASLR costs a leak; SMEP costs a stack-pivot into a kernel gadget chain. The exploit gets longer; it does not get impossible.

## Access control — three different worldviews

Above the hardware, the kernel has to decide *who can do what*. There are three classical models, all in production today, often layered:

### Discretionary access control (DAC)

The Unix model. Each object has an owner, and the owner decides who else gets in. Ten bits per file: `rwx` for owner, group, other.

```bash
$ ls -l /etc/shadow
-rw-r----- 1 root shadow 1872 Apr 12 03:04 /etc/shadow
```

Discretionary because the *user* decides; the kernel only enforces. DAC is fast, simple, and *useless* against a compromised user account. If the attacker is root, the model is silent.

### Mandatory access control (MAC)

A second, system-wide policy on top of DAC that *neither* the user *nor* root can override. SELinux and AppArmor on Linux; integrity levels and AppContainer on Windows. The policy is set by an administrator; even a process running as root is denied if the MAC label says so.

<Terminal title="selinux — refusing root">
{`# even as root, the policy says no
$ id
uid=0(root) gid=0(root)

$ cat /etc/shadow
cat: /etc/shadow: Permission denied

$ ausearch -m AVC -ts recent | tail -3
type=AVC msg=audit(...) avc:  denied  { read } for  pid=4421
  scontext=system_u:system_r:httpd_t:s0
  tcontext=system_u:object_r:shadow_t:s0  tclass=file`}
</Terminal>

MAC is what stops a compromised webserver from reading `/etc/shadow` even though `/etc/shadow` is technically readable by root. It is also why a real engagement so often ends with *"we got root in `httpd_t` context — now what?"*

### Capabilities and ACLs

The middle path: instead of giant binary "you are root or you are not", break root's powers into **capabilities** (Linux) or fine-grained **ACL entries** (Windows). A process can have `CAP_NET_BIND_SERVICE` (bind to port 80) without having `CAP_SYS_ADMIN` (own the world).

```bash
$ getcap /usr/bin/ping
/usr/bin/ping cap_net_raw=ep

# attacker on a misconfigured box:
$ getcap -r / 2>/dev/null
/opt/legacy/snapshot  cap_dac_read_search,cap_sys_ptrace=ep
```

That `cap_sys_ptrace` line is a privilege-escalation finding: any process can be ptraced, including root processes. An operator finds these by walking the filesystem with `getcap -r /` and `find / -perm -4000` and noting which binaries should not have the powers they have.

## Isolation primitives — the modern obsession

For about twenty years the answer to "how do I run untrusted code safely" was *don't*. Then it was *VMs*. Now it is **process-level isolation primitives** baked into the kernel, used by container runtimes, browsers, and sandboxes.

The three workhorses on Linux are:

- **Namespaces.** `pid`, `mount`, `net`, `ipc`, `uts`, `user`, `cgroup`. A process inside a `pid` namespace sees a different `/proc`. A process inside a `net` namespace sees a different `lo`. Chained together they produce the illusion of a private machine — a container.
- **cgroups.** Resource accounting and limits. Not a security boundary by itself, but a denial-of-service boundary: the sandboxed process cannot starve the host.
- **seccomp-bpf.** A per-process syscall filter. The process declares "I will only ever call these 30 syscalls"; the kernel kills the process if it tries any of the others.

This is the layout of every well-engineered sandbox you've used in the last decade — Chrome's renderer, Docker's default profile, systemd's `ProtectSystem=strict`, Firefox's content process. The interesting question is always **what's the syscall filter** and **what capabilities did they drop**. That's where the bugs are.

## A practical operator's hardening checklist

Switching sides for a moment: when I'm hardening a host, this is the list I run, in order. It isn't comprehensive. It is the highest-leverage list.

1. **Boot integrity.** UEFI Secure Boot enabled, vendor keys only. `tpm2_pcrread sha256:0,1,2,4,7` to confirm measured boot is producing sane values.
2. **Kernel mitigations.** `CONFIG_SLAB_FREELIST_RANDOM=y`, `CONFIG_SLAB_FREELIST_HARDENED=y`, `CONFIG_FORTIFY_SOURCE=y`, `CONFIG_INIT_ON_ALLOC_DEFAULT_ON=y`. Check with `grep CONFIG_ /boot/config-$(uname -r)`.
3. **Mitigations at runtime.** `cat /sys/devices/system/cpu/vulnerabilities/*` — every entry should read *Mitigation:* not *Vulnerable*.
4. **MAC enforcing.** `getenforce` returns `Enforcing` (SELinux) or `aa-status` shows enforce-mode profiles (AppArmor). Do not run permissive in production.
5. **Setuid audit.** `find / -perm -4000 -type f 2>/dev/null` and reduce the list. Most distros ship with twenty-plus; you probably need five.
6. **Capabilities audit.** `getcap -r / 2>/dev/null`. Anything with `cap_sys_admin`, `cap_dac_read_search`, or `cap_sys_ptrace` is an LPE waiting to happen.
7. **Syscall reduction for services.** For each service, write a seccomp profile or use `systemd`'s `SystemCallFilter=@system-service` with explicit deny lists. Use `strace -c` during normal operation to derive the minimum set.
8. **Kernel modules.** `lsmod` and `modprobe.blacklist` everything you don't ship. Most third-party drivers are signed garbage; if you don't need it, don't load it.
9. **Update cadence.** `unattended-upgrades` or equivalent. Most LPEs are patched within days; the half-life of an n-day on a typical fleet is the gap between patch release and your patch window.
10. **Audit pipeline.** `auditd` rules for `execve`, `connect`, and `open` on sensitive paths, shipped to a central store. You will not catch the breach in real time; you want a forensic trail.

## What to keep in mind

Operating-system security is the only domain in computing where defence-in-depth is the *prevailing paradigm*, not the buzzword. Each ring, each mitigation, each filter assumes the thing above it has already been bypassed.

For an operator, that's the practical lesson: don't chase the perfect exploit. Chase **chains**. Each layer above is one bug, one info leak, one timing hint. For a defender, the same lesson, mirrored: don't chase the perfect mitigation. Stack the layers; the exploit gets *longer* — and longer exploits get *unreliable*, and unreliable exploits get *caught*.

> The kernel is not the wall. The kernel is the last of many walls. Build them all.
